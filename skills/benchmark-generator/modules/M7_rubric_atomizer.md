# M7 ŌĆö Rubric Atomizer

**Stage 5 of the v1.1 modular generator.** Reads
`drafts/<project>.candidates.jsonl` (for `row_plan`),
`drafts/<project>.curated_evidence.jsonl` (M2),
`drafts/<project>.claims.jsonl` (M3), and
`drafts/<project>.answers.jsonl` (M6). Writes
`drafts/<project>.rubrics.jsonl`.

## Purpose

Decompose each row's expected answer into atomized scoring units:
`required_atoms` (propositions the answer must convey) and
`forbidden_atoms` (specific wrong conclusions the answer must not
contain). The current generator writes mechanical `path:lines µśŠńż║’╝Üsnippet`
atoms (88-89% of the corpus); M7 produces semantically scoreable
propositions instead.

## Why this is a host-LLM step

A rubric atom is a one-sentence proposition with a typed match style.
Generating one mechanically from a snippet either produces verbatim
quotes (current failure) or string-replace placeholders. The host LLM
can read the M6 answer, the M3 claims, and the M2 evidence, and
produce propositions that an LLM judge can semantically check.

## Inputs

For each row, you receive:

- `row_plan` (layer, answerability, answer_type, axis2, axis3)
- M2 selected_evidence: `evidence_id` + `statement`
- M3 claims: `id` + `text` + `evidence_ids` + `kind`
- M6 `expected_answer`: the prose to atomize

## Output shape

```jsonc
{
  "case_id": "nvdla-v1_1-L1-031",
  "required_atoms": [
    {
      "id": "A1",
      "role": "conclusion",
      "statement": "NVDLA derives per-layer INT8 scale factors from TensorRT calibration runs.",
      "match_type": "semantic_fact",
      "evidence_ids": ["E1"],
      "weight": 2
    }
  ],
  "forbidden_atoms": []
}
```

For L3 rows, atoms may carry `depends_on: ["A1", ...]` to express the
chain that the L3 structural-gate check requires.

## Rules

### Propositional, not pointer

The `statement` field of every required_atom must be a complete
declarative sentence. It must NOT match the pattern
`^<path>:<lines>\s*µśŠńż║’╝Ü` and must NOT be a verbatim substring of any
M2 `statement` or any M3 claim `text`.

Bad:
```
"hw/Makefile:4-16 µśŠńż║’╝ÜHW_DIR := $(VORTEX_HOME)/hw ..."
```

Good:
```
"VX_config.h is generated by hw/Makefile via the config target, which
invokes scripts/gen_config.py with VX_config.vh as input."
```

### Role / match_type / weight by answer_type

Pick the role + match_type that matches the answer's shape. Lint accepts
any combo from the legal sets but warns on mismatches.

| answer_type | primary conclusion atom |
|---|---|
| `yes_no` | role=`conclusion`, match=`semantic_yes_no`, weight=2 |
| `fact_check` | role=`conclusion`, match=`semantic_yes_no` or `semantic_fact`, weight=2 |
| `mechanism` | role=`conclusion`, match=`semantic_reasoning` or `semantic_fact`, weight=2; supporting `reasoning` atoms weight=1 |
| `procedure` | role=`conclusion`, match=`semantic_reasoning`, weight=2; per-step `procedure_step` atoms weight=1 |
| `location` | role=`conclusion`, match=`path_or_symbol`, weight=2 |
| `comparison` | role=`conclusion`, match=`semantic_fact`, weight=2; `comparison_point` atoms weight=1 each |
| `synthesis` | role=`conclusion`, match=`semantic_fact`, weight=2; multiple `evidence_fact` atoms weight=1 |
| `negative` | role=`conclusion`, match=`semantic_fact`, weight=2; for `unanswerable_ambiguous` use `boundary` role for the disambiguation atom |

### Atom count

- L1: 1 conclusion atom. Add a supporting `evidence_fact` only if the
  row's claim explicitly relies on a second proposition.
- L2: 1 conclusion atom + 1 supporting atom (matching the second
  source's role).
- L3: 1 conclusion + 1ŌĆō2 `reasoning` atoms that carry `depends_on`
  pointing at earlier atoms (this is what satisfies
  `L3_NO_ATOM_CHAIN`).
- Never exceed 4 atoms per row.

### evidence_ids

Every atom's `evidence_ids` list must reference one or more
`evidence_id`s present in M2's curated_evidence for the same case_id.
Empty list is allowed only for the special unanswerable cases below.

### implicit_domain_knowledge requirement

If `axis3_reasoning` includes `implicit_domain_knowledge`, the rubric
must include at least one atom with `role == "reasoning"` whose
statement is NOT a verbatim substring of any M2 evidence `statement`
and NOT a verbatim substring of any M3 claim `text`. This satisfies
the `IMPLICIT_DK_VERBATIM_REASONING` lint rule from Ship 1.

Practical recipe: write the reasoning atom as the *inference step* that
combines what the evidence shows with what the reader must apply from
domain knowledge. Example:

```
evidence:  "VX_local_mem accepts a TAG_WIDTH parameter computed from
           $clog2(NUM_TAGS)."
reasoning atom: "Because TAG_WIDTH must accommodate the largest
                 outstanding request id and $clog2 rounds up to whole
                 bits, doubling NUM_TAGS only adds one bit of tag width
                 ŌĆö a sublinear area cost."
```

### Forbidden atoms

A forbidden atom names a *specific* wrong conclusion the answer must
not contain. Generic "ńŁöµĪłÕŻ░ń¦░Ķ»üµŹ«ńøĖÕÅŹ" is rejected (the validator
flags it).

Required for these row types:

| trigger | what the forbidden_atom must encode |
|---|---|
| `answer_type == "yes_no"` | The opposite yes/no answer in a specific form. Example: if conclusion is "dispatch returns -EINVAL on null req", forbidden is "dispatch proceeds to write registers when req is null". |
| `answer_type == "fact_check"` | Both directions of the wrong conclusion when ambiguous: "the answer confirms the claim" + "the answer falsely denies the claim", or just the wrong direction the false_premise sets up. |
| `answerability == "unanswerable_false_premise"` | The specific false premise from the query, verbatim if possible. The forbidden_atom says "the answer accepts the premise that <quote>". |
| `answerability == "unanswerable_missing_evidence"` | "The answer fabricates a concrete conclusion about <topic> despite the snapshot containing no supporting evidence." |
| `answerability == "unanswerable_ambiguous"` | "The answer picks one of <candidates> without explicit caveat." |

`match_type` for forbidden atoms is almost always
`semantic_contradiction`; `severity` is `fatal` when failing the
forbidden costs the row outright, `major` when it only docks score.

## Few-shot exemplars

### Example A ŌĆö L1 / answerable / mechanism + long_tail + implicit_domain_knowledge

M2 evidence:
```

```


## Validation

```bash
python skills/benchmark-generator/scripts/validate_module_outputs.py \
  --module M7 \
  --candidates drafts/<project>.candidates.jsonl \
  --curated drafts/<project>.curated_evidence.jsonl \
  --claims drafts/<project>.claims.jsonl \
  --answers drafts/<project>.answers.jsonl \
  drafts/<project>.rubrics.jsonl
```

FAIL on:
- atom `statement` matches `^<path>:<lines>\s*µśŠńż║’╝Ü` pattern
- atom `statement` is a verbatim substring of any M2 statement or M3 claim text
- no conclusion atom present
- `evidence_ids` references an id not in M2 curated_evidence
- L3 row has no atom with non-empty `depends_on`
- yes_no/fact_check/false_premise/missing_evidence/ambiguous row has no forbidden_atom
- forbidden_atom statement is the generic boilerplate "ńŁöµĪłÕŻ░ń¦░ X µö»µīüõĖÄÕ╝Ģńö©Ķ»üµŹ«ńøĖÕÅŹńÜäń╗ōĶ«║"
- implicit_domain_knowledge claim without a non-verbatim reasoning atom

WARN on:
- more than 4 atoms in a row
- match_type mismatched against answer_type (e.g., yes_no row with no `semantic_yes_no` atom)
